# BACKLOG

- Agent status heuristics still conflate “assistant asked a question in prose” with “assistant is blocked on user input.” Tighten waiting detection to explicit approval/selection/input states instead of broad natural-language question patterns, or add recency/priority rules so older question text cannot dominate active work.
- Codex status detection is still regex-coupled to transient CLI chrome. The live Codex pane format has already drifted past the fixture set once, which let `Unknown` and stale preview behavior slip through. Add a lightweight workflow for capturing fresh real-pane samples per supported agent/version and refreshing `testdata/fixtures/` before releases.
- Align the documented/dashboard keybindings with the actual shipped TUI behavior. `README.md` presents `b`, `D`, `r`, `/`, `m`, `R`, and `s` as active actions, but [`internal/tui/app.go`](/home/ramtinj/personal-workspace/Colosseum/internal/tui/app.go) currently routes them to `unavailableFeatureStatus(...)`, and [`internal/tui/dialog/help.go`](/home/ramtinj/personal-workspace/Colosseum/internal/tui/dialog/help.go) lists them as unavailable. Either implement the actions or narrow the product surface in the docs/help copy.
- Status refresh currently has two authorities: the background `status.Poller` started in [`cmd/colosseum/main.go`](/home/ramtinj/personal-workspace/Colosseum/cmd/colosseum/main.go) and the direct `status.RefreshWorkspaceStatuses(...)` call in [`internal/tui/app.go`](/home/ramtinj/personal-workspace/Colosseum/internal/tui/app.go). Consolidate around one status-update path so tmux polling cadence, persisted status writes, and TUI message flow stay predictable.
- Decide whether Colosseum is a tmux session launcher or a real git-worktree manager, then align the product and code. The repo README and CLI long description promise “parallel AI coding agents across git worktrees,” and `new` collects a `--branch` value, but the current create path only opens a tmux session in an existing directory and persists the branch label in metadata. If worktrees are the product, add explicit `git worktree add` / branch lifecycle management; if not, remove that language from the UX and docs.
- Narrow or justify the “terminal-agnostic” positioning. The repo README and `context.md` describe Colosseum as terminal-agnostic, but the implementation is tightly coupled to tmux session lifecycle, tmux pane capture, tmux key bindings, and tmux client switching. Either redefine the product as tmux-native or introduce an abstraction boundary that could realistically support a second terminal backend.
- Split the Cobra surface before adding more commands. [`cmd/colosseum/main.go`](/home/ramtinj/personal-workspace/Colosseum/cmd/colosseum/main.go) currently owns the root command, all four subcommands, flag wiring, state/bootstrap helpers, and dashboard startup. That is still manageable with `new/list/attach/delete`, but it will turn into a maintenance bottleneck once `broadcast`, `diff`, config loading, or additional hooks land. Move toward a `cmd/` package with one file per subcommand and shared bootstrap helpers so future CLI growth does not stay coupled to one monolithic entrypoint.
- Session-prefix ownership is split between [`cmd/colosseum/main.go`](/home/ramtinj/personal-workspace/Colosseum/cmd/colosseum/main.go) and [`internal/tmux/session.go`](/home/ramtinj/personal-workspace/Colosseum/internal/tmux/session.go). `workspace.Manager` persists `SessionName` from its own prefix while tmux operations use the client’s prefix, so a future change to either constant can silently desynchronize stored metadata from real tmux session names. Collapse prefix configuration to one authority or derive stored names from the tmux client.
