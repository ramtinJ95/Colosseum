# BACKLOG

- Agent status heuristics still conflate "assistant asked a question in prose" with "assistant is blocked on user input." `claude.go:25` has broad NLP patterns like `(?i)(do you want|would you like|should i|shall i)`, and `detector.go` has a `questionEndsLine` (`\?\s*$`) pattern matching any line ending in `?` within a 3-line window. Tighten waiting detection to explicit approval/selection/input states instead of broad natural-language question patterns.
- Codex status detection is still regex-coupled to transient CLI chrome. The live Codex pane format has already drifted past the fixture set once, which let `Unknown` and stale preview behavior slip through. Add a lightweight workflow for capturing fresh real-pane samples per supported agent/version and refreshing `testdata/fixtures/` before releases.
- Align the documented/dashboard keybindings with the actual shipped TUI behavior. `b` (broadcast) is now implemented, but `README.md` still presents `D`, `r`, `/`, `m`, `R`, and `s` as active actions while [`internal/tui/app.go`](/home/ramtinj/personal-workspace/Colosseum/internal/tui/app.go) routes them to `unavailableFeatureStatus(...)`. Either implement the remaining actions or remove them from the README.
- Make keybinding help truly config-driven. Even after configurable `keys.*` support landed, [`internal/tui/sidebar/view.go`](/home/ramtinj/personal-workspace/Colosseum/internal/tui/sidebar/view.go), [`internal/tui/dialog/help.go`](/home/ramtinj/personal-workspace/Colosseum/internal/tui/dialog/help.go), and [`internal/tui/preview/view.go`](/home/ramtinj/personal-workspace/Colosseum/internal/tui/preview/view.go) still render literal shortcuts like `j/k`, `h/l`, and `n`, so remapped keys can work while the on-screen help lies. Thread the effective key bindings into help rendering or generate help text from the existing `KeyMap`.
- Thread the app keymap into child dialogs instead of hardcoding local shortcuts. [`internal/tui/dialog/delete.go`](/home/ramtinj/personal-workspace/Colosseum/internal/tui/dialog/delete.go), [`internal/tui/dialog/new_workspace.go`](/home/ramtinj/personal-workspace/Colosseum/internal/tui/dialog/new_workspace.go), and [`internal/tui/dialog/broadcast.go`](/home/ramtinj/personal-workspace/Colosseum/internal/tui/dialog/broadcast.go) match raw strings like `y`, `n`, `h`, `l`, `j`, `k`, and `ctrl+s`, so app-level remaps only affect the parent model. Promote dialog-specific bindings into a shared/config-derived keymap and use `key.Matches` consistently across parent and child models.
- Validate or deconflict duplicate key bindings in config. `internal/tui/app.go` resolves actions by switch order, so if a user assigns the same key to navigation and another action, the earlier case silently wins and the remap appears broken. Reject duplicates during config load or centralize dispatch so conflicts are surfaced instead of shadowed.
- Decide whether Colosseum is a tmux session launcher or a real git-worktree manager, then align the product and code. The repo README and CLI long description promise "parallel AI coding agents across git worktrees," and `new` collects a `--branch` value, but the current create path only opens a tmux session in an existing directory and persists the branch label in metadata. If worktrees are the product, add explicit `git worktree add` / branch lifecycle management; if not, remove that language from the UX and docs.
- Narrow or justify the "terminal-agnostic" positioning. The repo README and `context.md` describe Colosseum as terminal-agnostic, but the implementation is tightly coupled to tmux session lifecycle, tmux pane capture, tmux key bindings, and tmux client switching. Either redefine the product as tmux-native or introduce an abstraction boundary that could realistically support a second terminal backend.
- Split the Cobra surface before adding more commands. [`cmd/colosseum/main.go`](/home/ramtinj/personal-workspace/Colosseum/cmd/colosseum/main.go) currently owns the root command, all subcommands, flag wiring, state/bootstrap helpers, and dashboard startup. Move toward a `cmd/` package with one file per subcommand and shared bootstrap helpers so future CLI growth does not stay coupled to one monolithic entrypoint.
- Unify tmux session identity before expanding workspace lifecycle. [`internal/workspace/manager.go`](/home/ramtinj/personal-workspace/Colosseum/internal/workspace/manager.go) persists `Workspace.SessionName` as `<prefix><title>`, but all create/split/kill/switch calls still pass bare `Title` and rely on [`internal/tmux/session.go`](/home/ramtinj/personal-workspace/Colosseum/internal/tmux/session.go) to re-apply the prefix. That makes `SessionName` partially decorative today and raises drift risk if worktree creation, rename, import, or external-session reconciliation start trusting the stored value. Pick one authority for session identity and route all lifecycle code through it.
